# Summary-of-building-enterprise-customer-service-agent-using-Dify

使用Dify+Ragflow+Mem0搭建企业智能客服Agent的经验总结,Dify的具体编排可以看:企业内部客服-whole-workflow.jpeg

---

## 前序

**1、快速验证原型：** 在企业进行Agent搭建时，为了快速验证想法到可演示原型的转化，像Dify、Coze等低代码平台不需要投入大量研发资源搭建基础架构，非常适合快速验证Agent的业务价值。Dify得益于可视化编排、内置丰富的节点生态、嵌入式网页部署等功能，能够搭建出一个大致满足功能的Agent，在我们团队内部进行智能体开发时都先使用Dify进行一个智能体的大致编排出核心的模块。

**2、Dify的局限性：** 当然Dify也有不少缺点，对于日益复杂的业务结点会使得整个workflow非常难以维护，追踪链路也异常繁琐，只适合于简单Agent的搭建和复杂Agent的前期原型搭建,对于一个真正可落地于复杂场景的agent,需要编排非常多的节点，例如:意图识别模块、记忆模块、知识库检索模块、链路追踪模块、系统评估模块、具体与企业内部的OA、ERP、CRM等交互的业务模块，每一个模块都会有很多节点构成复杂的图，这时候更考虑Langchain、Langgraph等框架更加适合。

**3、写作目的：** 网上对于Dify复杂流程的搭建分享比较少，所以我想通过我的一些使用经验给出一点参考，帮助那些目前企业内部想要使用Dify进行快速原型验证的开发者能够快速学习大致的搭建流程，如果有错误不足的地方，还请各位批评指正。

## 概述

本文基于当前 Dify Chatflow 配置与代码节点，整理企业内部客服的端到端流程。重点覆盖意图识别+反问机制、意图路由、RAGFlow 集成、长期/短期记忆管理等核心环节，Dify的搭建图只列举了核心节点，实际生产环境中，需要考虑到Http请求调用异常、超时、重试机制等保证流程的正确，碍于篇幅，这些内容没有在流程图中体现。

---

## 1. 总体流程概览

整体路径可概括为：

1. 用户输入进入工作流
2. 意图识别（LLM 结构化识别）
3. 需要则触发反问澄清
4. 进入检索链路（RAGFlow + mem0 长期记忆检索）
5. 上下文组装（历史摘要 + 长期记忆 + 知识库）
6. 主模型生成回复
7. 记忆更新（短期摘要 + mem0 存储）

---

## 2. 意图识别与反问机制

意图识别模块是智能体中核心的模块之一，关系,在现实场景中，业务需求往往远比预想的更为复杂。例如，如何依据多轮对话内容准确判断意图和槽位，如何合理切分不同意图的对话内容，避免其在模型识别过程中相互干扰以及当业务槽位参数缺失如何进行反问补充等问题，都极具挑战性，所以要构造一个意图识别模块会有复杂的业务流程，在此Dify实现中，只实现了最基本的流程。

下面是Dify的意图识别的节点设置:

### Phase 1: 前置关键词快速检测

- 检测 greeting/thanks/bye/chitchat/handoff 等简单意图
- 命中则直接返回，无需走完整流程
- 输出：`intent_type` + `direct_answer`

### Phase 2: 意图 Case RAG 召回（合并高置信度直接回答检查）

- **单次检索**：调用 `search_intent_cases()` 召回 top-K 相似 Case
- **高置信度判断**：如果 top-1 满足阈值且有 `direct_answer`，直接返回
- **否则**：将召回的 Case 作为 Few-shot 示例进入 Phase 3

### Phase 3: 单次 LLM 综合识别

- **输入**：用户问题 + 历史对话 + RAG 召回的 Case 示例
- **Few-shot 示例格式**：
  ```
  【历史提问】...
  【最新提问】...
  【思考过程】...
  【处理】意图路由/直接回答
  【意图】business → M01 HR人事
  【槽位】{...}
  【分类】[...]
  ```
- **输出结构化字段**：
  - `intent_type`: 意图类型（greeting/thanks/chitchat/bye/handoff/business）
  - `clarity`: clear / vague
  - `clarity_score_total`: 清晰度评分（0-15）
  - `identified_module` / `identified_module_name`: 识别的业务模块
  - `extracted_slots`: 提取的槽位参数
  - `rewritten_query`: 改写后的检索句
  - `categories`: RAG 路由分类
  - `handling_mode`: intent_route / direct_answer
  - `direct_answer`: 直接回答内容（如有）

### 2.2 意图清晰度判断

- 通过 LLM 判断问题是否足够明确，输出 JSON：
  - `clarity`: clear / vague
  - `clarity_score`: 模块定位、问题描述、上下文信息三项评分
  - `identified_module` / `identified_module_name`
  - `missing_info`: 缺失信息列表
  - `possible_modules`: 候选意图（含置信度）

### 2.3 反问机制

- **条件**：`clarity == vague` 且 `clarification_round < 2`
- **反问生成规则**：
  - 将可能意图转换为用户可选项
  - emoji 编号
  - 语气友好，避免盘问感
- **流程**：
  - 反问完成后 `clarification_round += 1`
  - 直接返回反问内容给用户
  - 用户回复后再次进入意图识别
- **退出条件**：问题已清晰 或 反问次数达到上限

## 4. RAGFlow 集成

**1、传统文档切割的痛点：** 知识库检索RAG模块同样是智能体搭建的核心模块之一，由于企业内部文档数量和结构繁杂，涉及到Excle、Word、PDF、Email、PPT、TXT、Md等类型并且结构各不相同的文档，如果使用传统的文档切割策略:按段落切割、按结构切割、按语义切割、递归字符切割、滑动窗口切割等，都无法很好的适配所有的文档，并且需要自己可视化切割效果、测试召回率等一系列问题。

**2、RAGFlow的解决方案：** Ragflow正是为了解决以上痛点，它将文档按照通用、手册、书籍、论文、QA、PPT、邮件、简历等格式进行切割并且提供可视化效果，将企业内部的文档按照特定格式切割分别存储在不同的知识库中，向量数据库的元数据会携带不同的知识库id，只要知识库都是使用同一款向量数据库就能够同时对不同的知识库进行向量检索。在实际用户对话请求中，通常一句话需要跨多个知识库进行检索，结合前面的意图识别路由到不同的知识库，对多知识库进行召回文档就成为了成熟的解决方案。

**3、集成方式与PDF解析优化：** Ragflow提供了HTTP调用、MCP方式能够非常方便的集成到系统中，详细介绍由于篇幅不便展开，可以自行查看官方文档https://ragflow.io/docs 进行查阅,官方的文档解析器deepdoc对PDF的解析效果在实测中不如Mineru解析器，这是一款开源的PDF解析框架，同样，Ragflow也能集成Mineru作为PDF解析器，但是集成需要自部署Mineru,如果觉得繁琐可以调用官方免费提供的API，具体做法可以参考我另一个仓库：https://github.com/Dawn-hong/ragflow-plus.git

**4、检索评估的重要性：** 对知识库检索的评估同样非常重要，这影响到系统能否动态根据用户的问题案例提高文档的召回率和准确率，因此，在Ragflow中，我们对检索结果进行了评估，包括：
- 召回率（Recall）：检索结果中包含所有相关文档的比例。
- 准确率（Precision）：检索结果中正确文档的比例。
- F1 分数（F1 Score）：召回率和准确率的调和平均值。

这些可以通过RagAS框架进行实现，只要对整个Agent的流程进行链路追踪，记录用户问题和对于知识库召回的结果，就能自动化进行评估分数，根据具体的业务场景制定细粒度的评分机制，当阈值低于某值时，聚集低分样本，去进行更新排查文档、优化提示词、编排流程等提高整个Agent的回复质量。

下面是Dify的HTTP请求Ragflow的节点设置:
### 4.1 分类到知识库映射

RAGFlow 节点维护分类到知识库 ID 的映射表，例如：

- 薪酬与社保 → 薪酬/社保知识库
- IT服务台 → IT FAQ / Helpdesk 知识库
- 财务报销 → 报销制度知识库

### 4.2 调用检索接口

- 请求参数：
  - `dataset_ids`：本次检索使用的知识库列表
  - `question`：改写后的 query
  - `rerank_id`：重排序模型
  - `top_k` / `page_size` / `similarity_threshold`
- 结果解析：
  - 取回 `chunks`，拼接为 LLM 可读的 RAG 内容块
  - 标注来源文档与相似度

## 5. 长期与短期记忆管理

### 5.1 长期记忆（mem0）

**检索：**
- 结合历史摘要 + 最近对话拼接检索 query。
- 从 mem0 拉取用户画像、偏好、历史事实。

**存储：**
- 每轮对话结束后，将用户问题 + 助手回复写入 mem0。
- 用于跨会话记忆持久化。

### 5.2 短期记忆（会话内）

**短期历史获取：**
- 拉取最近 N 条消息，作为短期上下文。

**摘要机制：**
- 超过轮数或 token 阈值时，触发摘要生成。
- 摘要输出为结构化 JSON，包含：
  - `summary`（高密度摘要）
  - `facts`（长期有效事实）
  - `open_loops`（未解决问题）
  - `decisions`（已确认结论）
  - `todo_next`（下一步动作）
- 摘要写回会话变量，用于后续上下文补足。

## 6. 上下文组装策略

最终送入主 LLM 的上下文由四部分组成：

1. 历史摘要（短期记忆-旧）
2. 长期记忆（mem0）
3. RAGFlow 检索结果
4. 近期对话记录（短期记忆-新）

此结构保证：
- 优先有事实依据
- 同时保留短期对话连贯性
- 防止长对话导致 token 超限

`assembled_context` 包含：
- 知识库检索结果（RAGFlow chunks）
- 对话历史（recent_history）
- 长期记忆（long_term_memory）
- 历史摘要（conversation_summary）
- 已按优先级组装并裁剪

## 7. 主模型回复生成

### 7.1 System Prompt 角色定义

```
# 角色
你是公司内部客服助手，根据知识库提供准确、友好的解答。
```

### 7.2 回答原则

| 原则 | 说明 |
|------|------|
| **准确优先** | 严格基于提供的知识库内容回答，不编造 |
| **坦诚边界** | 知识库无法覆盖的问题，明确告知并建议联系对应部门 |
| **结构清晰** | 复杂问题用分步骤/分要点回答 |
| **语气友善** | 保持专业、亲切，可适当使用 emoji |
| **引用来源** | 回答末尾如有明确来源文件，可简要标注 |

### 7.3 上下文使用优先级

1. **知识库检索结果**（最高优先级，回答必须基于此）
2. **当前用户提问**
3. **对话历史**（辅助理解上下文）
4. **员工长期记忆**（仅作背景参考）

### 7.4 特殊情况处理

| 场景 | 处理方式 |
|------|----------|
| 知识库检索结果为空 | "抱歉，目前知识库中没有找到相关信息，建议您联系 [对应部门] 进一步咨询。" |
| 涉及敏感信息（薪资、个人隐私等） | 提醒用户通过安全渠道或线下咨询 |
| 问题超出职责范围 | 礼貌说明并指引正确联系方式 |

### 7.5 User Prompt 输入格式

```
## 员工提问
{query}

## 识别模块
{identified_module_name}

## 综合上下文
{assembled_context}

请基于以上信息，为员工提供准确、友好的回答。
```

## 8. 关键变量与状态

会话变量（Conversation Variables）：

- `recent_history`: 近期对话历史
- `long_term_memory`: mem0 长期记忆
- `conversation_summary`: 历史摘要
- `clarity`: 问题清晰度
- `clarification_round`: 反问次数
- `intent_case_matches`: RAG 召回的 Case 列表
- `extracted_slots`: 提取的槽位

环境变量：

- `mem0_base_url`
- `mem0_api_key`
- `qwen_api_key`
- `ragflow_base_url`
- `ragflow_api_key`

## 9. 典型执行路径示例

- 用户提问「我想申请年假」
- 意图 = HR人事
- clarity = clear → 不触发反问
- RAGFlow 检索 HR 知识库
- mem0 检索历史偏好
- 组装上下文 → 主模型回答
- 写入 mem0 + 更新摘要
